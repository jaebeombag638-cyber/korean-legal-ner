## 4. JSON 스키마 (어노테이션 포맷)

### 4.1 전체 구조

```json
{
  "info": { ... },
  "sections": [ { ... }, ... ],
  "annotations": [ { ... }, ... ]
}
```

### 4.2 info 필드

| 필드명 | 타입 | 필수 | 설명 | 형식/범위 |
|--------|------|------|------|---------|
| `dataType` | string | Y | 문서 종류 (판결문/심결례) | — |
| `caseNm` | string | Y | 사건명 | — |
| `caseTitle` | string | Y | 사건 제목 | — |
| `courtType` | string | Y | 심판 유형 | `"대법원"` 또는 `"하급심"` |
| `courtNm` | string | Y | 법원명 | — |
| `judmnAdjuDe` | string | Y | 판결 선고일 | `YYYY-MM-DD` |
| `caseNoID` | string | Y | DB 관리용 ID | — |
| `caseNo` | string | Y | 사건 번호 | — |
| `jdgmn` | string | N | 판시사항 | — |
| `keyword_tag.keyword` | array | Y | 키워드 배열 | 10개 이하 |
| `Reference_info.reference_rules` | string | N | 참조 조문 | — |
| `Reference_info.reference_court_case` | string | N | 참조 판례 | — |
| `caseClass` | string | Y | 사건 분류 | 민사/가사/형사A 등 |
| `annotation_count` | number | Y | annotation 총 개수 | — |

### 4.3 sections 배열 요소

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| `section_id` | string | Y | 섹션 ID, 형식: `^S\d+$` (S1, S2, ...) |
| `title` | string | Y | 섹션 제목 (판시사항, 이 유, 주 문 등) |
| `text` | string | Y | 섹션 본문 (원문 텍스트) |
| `char_count` | int | Y | 섹션 글자수 |

### 4.4 annotations 배열 요소

| 필드명 | 타입 | 필수 | 설명 | 형식/범위 |
|--------|------|------|------|---------|
| `annotation_id` | string | Y | 라벨 ID | `^A\d+$` (A1, A2, ...) |
| `section_id` | **array** | Y | 라벨 대상 섹션 ID 배열 | `["S9"]` ← **배열 타입** |
| `span` | array | Y | 텍스트 위치 정보 배열 | — |
| `span[].start` | int | Y | 라벨 시작 위치 (inclusive) | 섹션 내 상대 위치 |
| `span[].end` | int | Y | 라벨 끝 위치 (exclusive) | 섹션 내 상대 위치 |
| `original_text` | string | Y | 원본 텍스트 | `text[start:end]`와 일치 |
| `method` | string | Y | 비식별화 방식 | `"GENERALIZE"` 또는 `"TOKENIZE"` |
| `replacement_text` | string | Y | 대체 텍스트 | — |
| `link_entity_id` | string | Y | 동일 개체 연결 ID (PPRL용) | `^E-\d+$` (E-250844 등) |
| `rationale` | string | Y | 비식별화 처리 이유 | — |
| `rules_triggered` | string | Y | 비식별화 규칙 ID | `^R\d+$`, 범위: R1~R10 |

> **⚠️ 중요**: 공식 스키마 표에서 `section_id` 타입을 `string`으로 표기하지만,  
> 실제 데이터(JSON 예시)에서는 **배열** `["S9"]`로 저장되어 있음. 실제 데이터를 따를 것.

> **⚠️ 중요**: `span.end`는 **exclusive** (Python `text[start:end]` 슬라이스 기준).  
> `original_text`의 길이 = `end - start`

## 5. 익명화 규칙 (R1~R10)

공식 문서에 명시된 규칙. R5, R6, R9, R10의 구체적 내용은 공식 문서에 미공개.

> **⚠️ 실제 사용 범위**: PDF 가공 기준 섹션에 "규칙 ID(**R1~R8**)에 따라 정규식 및 NER 엔진을 활용하여 명시적 개인정보 탐지"라고 명시. 스키마 표기는 R1~R10이지만 실제 가공에서는 **R1~R8만 사용**. R9, R10은 예약 번호이거나 미적용 규칙일 가능성 높음 — EDA로 실제 데이터에 등장하는지 반드시 확인.
>
> **✅ EDA 확인 완료 (2026-07-14)**: 공개 데이터(라벨링데이터, 18,900개 파일, annotation 341,355건) 전수 조사 결과 **R1·R3·R7만 실제로 존재**. R2·R4·R8은 물론 R5·R6·R9·R10도 **0건**. 즉 공개 배포본에는 인명·주소·기관명만 라벨링되어 있고 날짜·주민번호·사업자번호는 애초에 학습 데이터가 없음 (자세한 수치는 `계획.md`/`진행일지.md` 1단계 참조).

| 규칙 | 점검 대상 | 탐지 패턴/방법 | 처리 방법 | 예외 |
|------|-----------|--------------|---------|------|
| **R1** | 성명(실명) | 정규식: `[가-힣]{2,4}` + 인명사전 매칭 | 소송상 지위로 대체 (증인-1, 피해자-1 등) | **사법인물(판사·검사·변호사) 제외** |
| **R2** | 주민등록번호 | 정규식: `\d{6}-\d{7}` | 완전 삭제 또는 `XXXXXX-XXXXXXX` 치환 | — |
| **R3** | 상세 주소 | 도로명/지번 패턴, 동·리·번지 포함 | 시·군·구 수준으로 일반화 + 이하 알파벳 대체 | — |
| **R4** | 정확한 사건 일자 | `YYYY.MM.DD` 또는 `YYYY-MM-DD` | **케이스별 다름** (아래 참조) | 행정·선고 관련 날짜 제외 |
| **R5** | 미공개 | — | — | — |
| **R6** | 미공개 | — | — | — |
| **R7** | 기업명·법인명·기관명 | 상호 사전/등기부 매칭 | `조직-1`, `조직-2` 등으로 대체 | **국가명 제외** |
| **R8** | 사업자등록번호/법인번호 | 정규식: `\d{3}-\d{2}-\d{5}` | 삭제 | — |
| **R9** | 실제 미사용 가능성 높음 | — | — | — |
| **R10** | 실제 미사용 가능성 높음 | — | — | — |

### R4 날짜 처리 — 3가지 케이스
```
케이스 1: 행정·선고 관련 날짜    → 비식별 처리 제외 (원문 그대로)
케이스 2: 고민감 사건(자살 등)   → 연도 앞 2자리도 마스킹: 19XX. XX. XX.
케이스 3: 일반 사건 날짜         → 연도 4자리만 유지: 1961. XX. XX.
```
어떤 케이스인지는 `rationale` 필드에 설명되어 있음.

### R3 주소 일반화 예시
- `부산시 중구 광복동` → `부산광역시 중구 A`
- `서울시 강남구 역삼동` → `서울특별시 강남구 A`
- 같은 구 내 다른 동은 `B`, `C` 등으로 구분

### method 필드 값
- `"GENERALIZE"`: 범주화 (이름 → 역할, 기관 → 조직-N, 날짜 → XX 마스킹)
- `"TOKENIZE"`: 토큰화 (삭제/해시 처리)
- 실제 데이터에서 확인된 것은 전부 `"GENERALIZE"`. `"TOKENIZE"` 사용 여부는 EDA로 확인 필요.

## 9. 핵심 주의사항 요약

코드 작성 시 반드시 기억해야 할 것들:

1. **span은 섹션 기준 상대 위치** — `annotations[].section_id`의 섹션 `text` 내에서의 위치. 문서 전체 기준 아님.

2. **span.end는 exclusive** — `section_text[start:end]` = `original_text`. `end - start` = 글자 수.

3. **section_id는 배열** — `ann['section_id']`는 `["S9"]` 형태의 리스트. `ann['section_id'][0]`으로 접근하거나 for문으로 순회.

4. **규칙은 R1~R10 존재** — 공식 문서에 R5, R6, R9, R10의 구체적 내용 미공개. EDA 시 이 규칙들이 실제 데이터에 나오는지, 어떤 텍스트인지 반드시 확인.

5. **R1 사법인물 예외** — 판사·검사·변호사 이름은 R1 처리 대상에서 제외. `rationale` 필드 참조.

6. **R4 날짜 3-케이스** — 행정/선고 날짜는 유지, 고민감 사건은 연도도 마스킹, 일반 사건은 월일만 마스킹.

7. **R7 국가명 예외** — `한국`, `미국` 등 국가명은 조직-N 처리 대상 아님.

8. **method는 사실상 GENERALIZE 단일** — 실제 데이터에서 TOKENIZE가 나오는지 EDA로 확인 필요.

9. **link_entity_id** — 동일 개체의 여러 등장을 연결하는 ID (`E-\d+` 형식). NER 학습에는 불필요하지만 데이터 이해에 유용.

10. **라벨링데이터만 사용** — 원천데이터 폴더는 인코딩 깨짐 문제 있음.

11. **span-original_text 불일치 약 10.4%** (2026-07-14, 200파일/3,099span 샘플 검사) — `section_text[start:end] != original_text`인 annotation이 존재. 원문이 이미 `#이름#` 형태로 치환된 경우(6.5%)와 위치 자체가 어긋난 경우(3.6%)로 나뉨. 전처리 코드에서 `verify_span`을 assert가 아닌 필터링 조건으로 사용할 것 (계획.md 0단계/2단계 참조).
