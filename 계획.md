### JSON 구조 (핵심만)
```json
{
  "info": { "caseClass": "가사", "annotation_count": 14 },
  "sections": [
    { "section_id": "S9", "text": "...성립을 다퉈이 있다...", "char_count": 1759 }
  ],
  "annotations": [
    {
      "annotation_id": "A1",
      "section_id": ["S9"],
      "span": [{ "start": 259, "end": 262 }],
      "original_text": "최도선",
      "method": "GENERALIZE",
      "replacement_text": "증인-1",
      "link_entity_id": "E-250844",
      "rationale": "사건 인물 실명은 직접식별자이므로 일반화 처리",
      "rules_triggered": "R1"
    }
  ]
}
``` 

> **데이터 구조 주의사항 3가지**
> 1. `span`은 **섹션 텍스트 내 상대 위치** (문서 전체 기준 아님)
> 2. `span.end`는 **exclusive** — `text[start:end]`가 `original_text`와 일치 (Python slice 기준)
> 3. `annotations[].section_id`는 **배열 타입** `["S9"]` (공식 스키마 표에는 string이라고 쓰여 있지만 실제 데이터는 배열)

### 2단계: 전처리 → BIO 변환 (3~4일)

**핵심 주의사항**:
- `span`이 섹션 내 상대 위치이므로, 섹션별로 독립 처리한다
- `annotations[].section_id`는 **배열**(`["S9"]`)이므로 리스트 처리 필수
- `span.end`는 exclusive (end 위치 문자는 포함 안 됨)

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("klue/bert-base")

# NER 레이블 정의 — 1단계 EDA로 확정 (R1·R3·R7만 실제 존재, DAT/ID는 학습 데이터 0건이라 제외)
LABELS = ["O", "B-PER", "I-PER", "B-ORG", "I-ORG", "B-LOC", "I-LOC"]
label2id = {l: i for i, l in enumerate(LABELS)}
id2label = {i: l for i, l in enumerate(LABELS)}

# 규칙 → NER 태그 매핑 (1단계 EDA로 확정. R2/R4/R5/R6/R8/R9/R10은 데이터에 없어 매핑 자체가 불필요)
RULE_TO_TAG = {
    "R1": "PER",
    "R3": "LOC",
    "R7": "ORG",
}


def build_section_annotation_map(data):
    """JSON 1개에서 section_id → [annotation 리스트] 매핑 생성.
    section_id가 배열 타입이므로 순회 처리.
    """
    mapping = {s['section_id']: [] for s in data['sections']}
    for ann in data.get('annotations', []):
        for sid in ann['section_id']:   # 배열이므로 for문
            if sid in mapping:
                mapping[sid].append(ann)
    return mapping


def is_drift_annotation(section_text, ann):
    """
    span 불일치 중 'drift'(위치 자체가 다른 텍스트를 가리킴)만 True를 반환한다.
    'placeholder' 불일치(원문이 이미 #이름# 등으로 치환된 경우)는 위치 자체는 맞으므로
    False를 반환해 학습에 그대로 사용한다 (1단계 EDA 결정, 진행일지.md 참조).
    """
    for span in ann['span']:
        extracted = section_text[span['start']:span['end']]
        orig = ann['original_text']
        if extracted == orig:
            continue  # 완전 일치
        if '#' in orig or '#' in extracted:
            continue  # placeholder 케이스 — 위치는 신뢰, 그대로 사용
        return True  # drift — 위치 자체가 틀림, annotation 전체를 제외
    return False


def section_to_bio(section_text, annotations_in_section):
    """
    섹션 텍스트 + 해당 섹션의 annotations → (token_ids, BIO 태그 리스트)

    span은 character 기준, BERT는 subword 토큰 기준
    → character offset → token offset 변환 필요
    """
    # 1. 문자 단위 레이블 초기화
    char_labels = ["O"] * len(section_text)

    # 2. annotation마다 해당 위치에 레이블 부여 (drift 불일치는 건너뜀)
    for ann in annotations_in_section:
        tag = RULE_TO_TAG.get(ann['rules_triggered'], "O")
        if tag == "O":
            continue  # 미정 규칙은 일단 O 처리
        if is_drift_annotation(section_text, ann):
            continue  # 위치 자체가 틀린 annotation은 라벨링에서 제외 (잘못된 라벨 주입 방지)
        for span in ann['span']:
            start, end = span['start'], span['end']
            char_labels[start] = f"B-{tag}"
            for i in range(start + 1, end):
                char_labels[i] = f"I-{tag}"

    # 3. 토크나이저로 분리 후 토큰 레벨 레이블 매핑
    # add_special_tokens=True: [CLS], [SEP] 포함 → BERT 사전학습 형식 유지
    # truncation=True, max_length=512: 법률 섹션이 512 초과하는 경우 많음 (실측 13.29%)
    #   → 512 이후 토큰의 annotation은 학습에서 누락됨 (6단계 한계점 참조)
    encoding = tokenizer(
        section_text,
        return_offsets_mapping=True,
        add_special_tokens=True,
        truncation=True,
        max_length=512,
    )
    token_labels = []
    for token_start, token_end in encoding['offset_mapping']:
        # [CLS], [SEP] 등 special token은 offset (0, 0) → loss 계산에서 제외
        if token_start == 0 and token_end == 0:
            token_labels.append(-100)
        else:
            token_labels.append(label2id[char_labels[token_start]])

    return encoding['input_ids'], token_labels


def verify_span(section_text, ann):
    """진단용(집계 전용): original_text가 span 위치에 정확히 있는지 확인.
    실제 필터링은 이 함수가 아니라 is_drift_annotation()이 담당한다 — 완전 일치가 아니어도
    placeholder 케이스는 학습에 포함시키기 때문. verify_span은 EDA 리포트용 통계 집계에만 쓴다.
    """
    for span in ann['span']:
        extracted = section_text[span['start']:span['end']]
        if extracted != ann['original_text']:
            return False
    return True
```

**검증 단계 (필수, 2026-07-14 기준 수정)**: 1단계 EDA에서 전체 341,355개 span 기준 불일치율을 **13.09%**로 확정 (원문 placeholder 치환 5.23% + 위치 어긋남 7.86%). 두 유형을 다르게 처리하기로 결정 (사용자 확인 완료, 진행일지.md 참조):
- **placeholder(5.23%)**: span 위치 자체는 정확히 개인정보 자리를 가리키므로 **그대로 라벨링에 사용**
- **drift(7.86%)**: 위치가 완전히 다른 텍스트를 가리켜서 그대로 쓰면 잘못된 라벨을 주입하게 되므로 **제외**

`is_drift_annotation()`이 이 구분을 구현한다. 실제로 필터링되어 학습에서 빠지는 비율은 **7.86%**뿐이고, 나머지 92.14%(placeholder 포함)는 그대로 사용한다.

**train/val/test 분리**:
- Training 폴더 → train 그대로 사용 (일반판결 12,000 / 1-2-최종심 4,800)
- Validation 폴더 → 절반씩 분리 (랜덤 시드 고정)
  - 일반판결: val 750 / test 750
  - 1-2-최종심: val 300 / test 300
- 분야 비율 유지한 stratified split 권장 (Validation 분리 시)

### 3단계: 모델 학습 (3~5일)

```python
from transformers import AutoModelForTokenClassification, TrainingArguments, Trainer

# LABELS, label2id, id2label — 2단계에서 이미 정의. 그대로 사용.

model = AutoModelForTokenClassification.from_pretrained(
    "klue/bert-base",
    num_labels=len(LABELS),
    label2id=label2id,
    id2label=id2label,
)

training_args = TrainingArguments(
    output_dir="./results",
    num_train_epochs=3,
    per_device_train_batch_size=16,
    per_device_eval_batch_size=32,
    learning_rate=3e-5,
    evaluation_strategy="epoch",
    save_strategy="epoch",
    load_best_model_at_end=True,
)

from transformers import DataCollatorForTokenClassification
```

### 5단계: 데모 앱 → Streamlit (2~3일)

**Streamlit 선택 이유**: Hugging Face Spaces에 무료 배포 가능, 링크 하나로 포트폴리오 제출 가능

```python
# app.py
import streamlit as st
from transformers import pipeline

@st.cache_resource
def load_model():
    return pipeline("ner", model="./best_model", aggregation_strategy="simple")

ner = load_model()

st.title("한국어 판결문 개인정보 익명화")
text = st.text_area("텍스트 입력", height=200)

if st.button("익명화 실행"):
    results = ner(text)
    # 탐지된 entity 하이라이트 + 교체 텍스트 표시
```

**기능**:
1. 텍스트 입력 → 탐지된 entity 색상별 하이라이트
2. 익명화된 결과 텍스트 출력
3. 탐지 entity 목록 표 (태그, 원문, 대체 텍스트)

### 6단계: 결론 정리 (1~2일)

노트북 마지막 섹션에 작성:

1. **성능 요약표**
   | Entity | Precision | Recall | F1 |
   |--------|-----------|--------|----|
   | PER | - | - | - |
   | ORG | - | - | - |
   | LOC | - | - | - |
   | 전체 | - | - | **목표 0.80** |

2. **공식 모델과 비교**
   - GPT-OSS-120B: F1 0.9589 (120B 파라미터, QLoRA) — AI Hub 비공개 test셋 기준
   - 본 모델: F1 ??? (110M 파라미터, 일반 파인튜닝) — Validation에서 직접 분리한 test셋 기준
   - ⚠️ 동일 test셋이 아니므로 수치 비교는 참고용이며, 포트폴리오에 조건 차이 명시 필요
   - 차이 원인 분석 (모델 크기, 학습 기법, test셋 차이 등)

3. **한계점**
   - 판결문 특화 표현이 일반 업무 문서와 다른 패턴 존재
   - R1 사법인물 예외 처리의 완성도
   - 512 토큰 초과 섹션 13.29%(실측): truncation으로 인해 해당 섹션 후반부 annotation 학습 누락
   - 태그 범위가 PER·LOC·ORG로 제한됨: 공개 데이터셋 자체에 날짜(R4)·주민번호(R2/R8) annotation이 없어 애초에 학습 불가능했음 (모델의 한계가 아니라 데이터의 한계)
   - span 불일치 13.09%를 필터링하고 학습해 원본 데이터의 약 13%를 활용하지 못함

4. **v2 방향** (이 프로젝트 이후 별도 태스크)
   - KoBERT 또는 KLUE-RoBERTa로 교체
   - 날짜·주민번호 등이 포함된 다른 데이터소스를 추가해 DAT/ID 태그 확장
