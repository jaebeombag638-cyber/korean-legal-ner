## 1단계 — EDA (탐색적 데이터 분석) (완료: 2026-07-14)

### EDA가 뭐고 왜 하는가
EDA(Exploratory Data Analysis)는 본격적으로 모델을 학습시키기 전에 **데이터가 실제로 어떻게 생겼는지 눈으로/숫자로 확인하는 단계**다. "문서에 이렇게 써있으니까 이럴 것이다"라고 가정하고 바로 코드를 짜면, 나중에 가정이 틀렸을 때 처음부터 다시 해야 한다. 그래서 18,900개 파일 **전체**를 실제로 열어서 세어본다.

노트북 파일: [`notebooks/01_eda.ipynb`](notebooks/01_eda.ipynb) — 아래 내용의 코드와 실행 결과가 그대로 들어있다.

### 한 일과 결과

**① 어떤 익명화 규칙(R1~R10)이 실제로 쓰였나?**

annotation 341,355건을 전부 세어본 결과:

| 규칙 | 의미 | 건수 | 비율 |
|------|------|------|------|
| R7 | 기관명 | 310,214 | 90.87% |
| R1 | 인명 | 19,477 | 5.71% |
| R3 | 주소 | 11,664 | 3.42% |
| R2, R4, R5, R6, R8, R9, R10 | 주민번호/날짜/기타 | **0** | **0%** |

**⚠️ 가장 중요한 발견**: 공식 문서(`official.md`)는 R1(인명)·R2(주민번호)·R3(주소)·R4(날짜)·R7(기관명)·R8(사업자번호)이 실제로 쓰인다고 했는데, **우리가 받은 공개 데이터에는 R1·R3·R7 세 개만 있고 나머지는 단 한 건도 없다.** README에 적힌 목표("이름·주소·날짜·기관명 등 자동 탐지")에서 **날짜(DAT)와 주민번호(ID)는 애초에 이 데이터로는 학습할 수 없다**는 뜻이다. AI Hub가 공개용으로 배포하면서 민감도 높은 항목(주민번호 등)을 아예 뺐을 가능성이 있다.

→ 이건 프로젝트 목표를 다시 정해야 할 만큼 중요한 발견이라 사용자에게 확인을 요청함.

**② `method`(비식별화 방식) 종류**

`GENERALIZE`(범주화, 예: 이름→"증인-1")만 100% 사용됨. `TOKENIZE`(삭제)는 한 번도 안 쓰임 → 전처리 코드를 단순하게 유지해도 됨.

**③ 사건 분야(`caseClass`) 분포**

| 분야 | 비율 (실제) | 비율 (공식 문서) |
|------|-----------|-----------------|
| 민사 | 40.28% | 28.33% |
| 일반행정 | 25.91% | 26.67% |
| 형사 | 14.18% | 20% (A/B 합산) |
| 근로자/금융조세/특허 | 각 3.57% | 각 5% |
| 가사 | 2.55% | 3.33% |
| 개인정보/기업 | 각 2.38% | 3.33% |
| 세무 (문서에 없던 항목) | 1.50% | — |

실제 비율이 공식 문서와 꽤 다르다. 나중에 train/val/test를 나눌 때는 이 **실측 비율**을 기준으로 삼아야 한다.

**④ span 위치가 실제 텍스트와 일치하는 비율 (전체 데이터 기준)**

- 일치: 296,687건 (86.91%)
- 불일치 - 원문이 이미 가려짐(`#이름#` 등, "placeholder" 유형): 17,838건 (5.23%)
- 불일치 - 위치 어긋남 ("drift" 유형): 26,830건 (7.86%)
- **불일치 합계: 13.09%**

0단계에서 200개 파일 샘플로 봤던 10.4%보다 실제로는 조금 더 높았다(13.09%).

**추가 결정 (2026-07-14, 사용자 확인)**: 처음엔 불일치 13.09%를 전부 버리려고 했는데, 두 유형이 사실 성격이 다르다는 걸 알게 됐다.

- **placeholder 유형**은 `span`이 가리키는 **위치 자체는 맞다.** 예를 들어 `original_text`가 `#이름#`(4글자)인데 실제로 그 위치엔 `#이름`(3글자)만 있는 식으로, 끝 글자 하나가 안 맞을 뿐 "여기에 개인정보가 있다"는 정보 자체는 정확하다. → 이건 버리지 않고 **위치를 그대로 믿고 라벨(PER/LOC/ORG)로 사용**하기로 함.
- **drift 유형**은 `span` 위치가 아예 딴 곳을 가리킨다. 예를 들어 `original_text`가 `"대법원"`인데 실제 그 위치엔 `" 보호"`처럼 전혀 상관없는 글자가 있다. 이걸 그대로 쓰면 평범한 텍스트에 "이건 기관명이다"라는 **틀린 정답**을 붙이는 셈이라, 모델을 오히려 헷갈리게 만든다. → 이것만 제외.

결과적으로 **실제로 버려지는 비율은 13.09%가 아니라 7.86%(drift만)**로 줄었다. `계획.md`의 `section_to_bio` 코드에 `is_drift_annotation()` 함수로 구현해뒀다.

**⑤ 섹션 텍스트 길이 (BERT가 한 번에 볼 수 있는 글자 수 제한 확인)**

BERT 계열 모델은 한 번에 최대 512"토큰"(대략 글자보다 조금 더 세밀한 단위)까지만 처리할 수 있다. 그보다 긴 섹션은 뒷부분이 잘려서, 그 안의 개인정보는 학습에 아예 반영되지 않는다.

- 전체 161,186개 섹션 중 512토큰 초과: **13.29%** (계획.md의 추정치 21%보다 낮았음 — 추정이 아니라 실측값으로 갱신)
- 평균 410.8 토큰, 중앙값 24 토큰 → 대부분 섹션은 짧고(제목, 주문 등), 일부(본문 "이 유" 섹션)만 아주 길다
- 최댓값 119,742토큰(원문 약 23만 자)인 문서가 있었는데, 확인해보니 `개인정보` 분야의 유출 사건 판결문으로, 유출된 개인정보 목록이 통째로 들어간 정상적인(버그 아닌) 극단적 사례였다.

**⑥ 문서 1개당 개인정보(annotation) 개수**

평균 18.06개, 중앙값 9개. 최댓값 2,380개는 마찬가지로 `개인정보` 분야 사건. 개인정보가 하나도 없는 문서도 256개(1.35%) 있었다 — 이런 문서는 "여기엔 개인정보가 없다"를 학습시키는 데 쓰인다.

### 결정된 것 (사용자 확인 완료)
- **태그셋**: `R2·R4·R5·R6·R8·R9·R10`이 데이터에 전혀 없으므로, NER 태그는 **PER·LOC·ORG 세 개로 확정** (DAT·ID는 목표에서 제외). `README.md`/`계획.md`에 반영 완료.
- **span 불일치 처리**: placeholder 유형(5.23%)은 위치를 믿고 그대로 사용, drift 유형(7.86%)만 제외. `계획.md`의 `is_drift_annotation()`으로 구현.

## 2단계 — 전처리: span → BIO 변환 (완료: 2026-07-14)

### 목표
1단계에서 정한 태그셋(PER·LOC·ORG)과 필터링 기준(drift만 제외)을 실제 코드로 만들어서, 글자 위치로 표시된 개인정보 정보를 BERT가 학습할 수 있는 "토큰별 정답 라벨" 형태로 바꾼다. 그리고 18,900개 파일 전체를 train/val/test로 나눈다.

### 한 일
1. **공용 코드 모듈 작성**: `scripts/ner_common.py`에 `section_to_bio`(핵심 변환 함수), `is_drift_annotation`(1단계에서 정한 필터링 기준), `build_section_annotation_map`을 실제 동작하는 코드로 옮겼다. 노트북과 스크립트가 같은 코드를 재사용하게 해서, 계획.md 안의 예시 코드와 실제 돌아가는 코드가 따로 놀지 않게 했다.
2. **train/val/test 분리**: AI Hub가 이미 나눠준 대로 Training(16,800건)은 전부 train으로, Validation(2,100건)은 분야(caseClass) 비율을 유지하면서 절반씩 val(1,048건)/test(1,052건)로 무작위 분리했다 (`stratified_half_split`, seed=42).
3. **전체 변환 실행** (`scripts/preprocess_full.py`): 18,900개 파일의 **섹션 하나하나(161,186개)**를 독립적인 학습 예제로 만들어 `data/processed/{train,val,test}.jsonl`에 저장했다. (`data/`는 git에서 제외되므로 이 결과물도 git에는 안 올라간다 — 필요하면 스크립트를 다시 돌려서 재생성하면 된다.)

### 결과
- 예제 수: train 143,292 / val 8,868 / test 9,026 (합계 161,186 — 1단계에서 센 전체 섹션 수와 정확히 일치)
- drift 필터링으로 실제 제외된 annotation: 24,102(train) + 1,861(val) + 867(test) = **26,830건** — 1단계 EDA에서 전체 데이터로 센 `mismatch_drift` 값(26,830)과 **정확히 일치**. 같은 숫자가 서로 다른 계산 경로에서 똑같이 나왔다는 건 코드가 맞게 동작한다는 신호다.
- **라벨 직접 검증**: `부산지방법원`이 `부산`(B-ORG)+`##지방`(I-ORG)+`##법`(I-ORG)+`##원`(I-ORG) 네 토큰으로 정확히 쪼개져서 라벨링됐고, `#이름#` placeholder도 `#`(B-PER)+`이름`(I-PER)로 위치를 믿고 그대로 라벨링된 걸 직접 눈으로 확인했다. `서울지방법원 남부지원`처럼 공백 낀 복합 기관명도 하나의 개체로 이어붙는 것까지 확인 — 함수가 의도대로 동작한다.

### 예상 밖의 발견 — 태그 간 심한 불균형
train 데이터의 개체 시작(`B-`) 토큰 수를 세어보니 **ORG 109,176개 : PER 4,151개 : LOC 4,465개**로, ORG가 나머지 두 태그를 합친 것보다도 10배 이상 많다. 1단계에서 R7(기관명)이 90.87%였던 것과 같은 맥락이다.

**왜 문제가 될 수 있나**: 이런 상태로 학습하면 모델이 "그냥 다 ORG라고 하면 점수가 잘 나오네" 하는 식으로 편향될 위험이 있다. 즉 ORG는 잘 맞히지만 PER·LOC는 상대적으로 놓칠 가능성이 높다.

→ 3단계 학습에서 class weight(태그별 가중치) 적용을 검토하고, 4단계 평가에서는 전체 F1만 보지 말고 **entity 태그별 F1을 반드시 따로** 확인해야 한다.

## 3단계 — 모델 학습 (완료: 2026-07-19)

### 목표
`data/processed/{train,val}.jsonl`을 이용해 `klue/bert-base`를 NER용으로 파인튜닝한다. 계획대로면 learning rate·epoch 조합 4회(Run1~Run4)를 실험해야 하지만, 이번엔 **Run1(baseline)만 먼저 완주**하기로 범위를 좁혔다 (아래 "작업 범위 결정" 참조).

### 한 일
1. **학습 스크립트 작성** (`scripts/train.py`): `AutoModelForTokenClassification`으로 klue/bert-base에 NER 분류 헤드를 얹고, `Trainer`로 학습한다. `--lr`/`--epochs`/`--class-weight` 인자로 계획.md의 Run1~Run4를 모두 같은 스크립트로 재현할 수 있게 만들었다 (`WeightedTrainer`가 2단계에서 발견한 ORG:PER:LOC 불균형 대응용 class weight를 담당, Run4용).
2. **평가용 노트북** (`notebooks/03_train.ipynb`): 스크립트 설명, 스모크 테스트 결과, Run1 실행 방법을 정리.
3. **파이프라인 검증**: 전체 학습(143,292건, 3 epoch, 수 시간 소요)을 돌리기 전에 train 앞부분 일부(64건)만으로 1 epoch 스모크 테스트를 돌려 학습→평가→저장이 에러 없이 도는지 확인했다.
   - 이 과정에서 `accelerate` 패키지가 이 프로젝트 파이썬 환경에 없어서 새로 설치해야 했다 (`Trainer`가 내부적으로 요구).
   - `transformers` 5.x부터 `TrainingArguments`의 `evaluation_strategy` 인자명이 `eval_strategy`로 바뀐 것도 확인.

### 작업 범위 결정 (2026-07-18)
4개 실험을 한 번에 다 돌리지 않고 **Run1(baseline, lr=3e-5, epoch=3)부터 먼저 완주**한다. 2,000건 샘플로 측정한 처리량 기준 전체 학습은 약 2.5~3시간 예상. 파이프라인이 정상 동작하고 체크포인트/메트릭이 저장되면 이 단위로 `practice` → PR → `main` 흐름을 진행하고, Run2~Run4는 후속 작업으로 미룬다.

### Git 작업 방식 문서화
"main에 Claude(AI) 기여 표시를 남기지 않는다"는 규칙이 그동안 어떤 프로젝트 문서에도 적혀 있지 않았던 걸 확인 — `계획.md`의 새 "작업 방식 (Git)" 섹션에 명시했다.

Run1(baseline, lr=3e-5, epoch=3) 학습을 완주했다. val set 기준:

|--------|-----------|--------|----|----|

**전체 micro F1 0.9071로 목표(0.80) 이미 초과 달성**했지만, entity별로 보면 2단계에서 예상했던 대로 **PER(F1 0.49)·LOC(F1 0.79)가 ORG(F1 0.93)보다 훨씬 약하다** — ORG가 support의 92.5%(7,045/7,615)를 차지해서 micro F1이 실제로는 ORG 성능에 크게 좌우된 결과다. class weight를 적용하는 Run4에서 PER·LOC가 개선되는지가 다음 관전 포인트.

### 결과 — Run2~Run4 완료 (2026-07-19, 자동 순차 실행)

Run1 완료 직후 계획대로 나머지 세 실험(Run2: lr=5e-5/epoch=5, Run3: lr=2e-5/epoch=3, Run4: lr=3e-5/epoch=3 + class weight)을 자동으로 순차 실행했다. val set micro F1 비교:

| 실험 | learning_rate | epoch | class weight | micro F1 | PER F1 | LOC F1 | ORG F1 |
|------|--------------|-------|--------------|---------|--------|--------|--------|
| Run1 | 3e-5 | 3 | X | 0.9071 | 0.49 | 0.79 | 0.93 |

| **Run3** | **2e-5** | **3** | X | **0.9077** | 0.47 | 0.79 | 0.93 |
| Run4 | 3e-5 | 3 | O | 0.8230 | 0.19 | 0.66 | 0.89 |

### 예상 밖의 발견 — class weight가 오히려 전체 성능을 깎아먹었다

Run4에 태그 빈도 역비례 class weight를 적용하면 PER·LOC의 **recall**은 오히려 오르지만(LOC recall 0.88→0.92, PER recall도 큰 차이 없음), **precision이 크게 무너진다** (PER precision 0.56→0.12, ORG precision 0.92→0.83). 결과적으로 모델이 "일단 PER·LOC로 더 자주 예측"하도록 편향되면서 오탐(false positive)이 급증했고, entity-level F1은 개선은커녕 **PER F1이 0.49→0.19로 오히려 절반 이하로 떨어졌다.** micro F1도 0.907대에서 0.823으로 크게 하락했다.

**해석**: 단순 역빈도 가중치(inverse frequency)는 "적게 등장하는 태그를 더 세게 처벌"하는 방식이라, 데이터가 원래 압도적으로 ORG 위주(26:1:1)인 상황에서는 과잉보정되어 모델이 전반적으로 개체를 과다 예측하게 만든 것으로 보인다. 클래스 불균형 대응은 "가중치를 걸면 무조건 좋아진다"는 게 아니라는 걸 실제로 확인한 사례 — 6단계 한계점에 반영할 내용.

### 최종 선택
Run1~Run3는 사실상 동급(F1 0.9071~0.9077, 차이가 통계적으로 유의미하지 않을 만큼 작음)이고, class weight(Run4)는 오히려 역효과였다. **Run3(lr=2e-5, epoch=3, micro F1 0.9077)를 최종 baseline 모델로 선정**해 4단계 평가에 사용한다. PER·LOC 성능 개선은 이번 4개 실험만으로는 달성하지 못했음을 6단계 한계점에 정직하게 기록한다.

## 3단계 추가 — Run5: class weight 재시도 (진행 중, 2026-07-22)

### 계기
4단계 평가(test F1 0.9045)까지 마친 뒤 프로젝트 전체를 다시 검토하다가, PER F1(0.55)이 계속 약한 채로 남아있는데 시도한 대응이 Run4(class weight) 단 하나뿐이었다는 점을 짚었다. Run4가 왜 실패했는지 좀 더 정확히 진단해보기로 했다.

### 원인 재분석
`data/processed/preprocess_stats.json`의 실제 train 토큰 카운트(O 16,486,956 / B-ORG 109,176 / I-ORG 237,307 / B-PER 4,151 / I-PER 5,971 / B-LOC 4,465 / I-LOC 18,646)로 Run4의 가중치 공식(`counts.sum()/(len(LABELS)*counts)`)을 직접 계산해봤다.

| 태그 | Run4 가중치(역산) |
|------|------|
| O | 0.146 |
| B-ORG | 22.1 |
| I-ORG | 10.2 |
| B-PER | **580.6** |
| I-PER | 403.6 |
| B-LOC | 539.6 |
| I-LOC | 129.2 |

O 토큰이 압도적으로 많다 보니(전체의 97.8%) O 가중치가 0.146까지 깎였다 — "아무 텍스트나 PER이라고 예측해도 손실이 거의 안 늘어나고, 진짜 PER 하나를 놓치면 손실이 580배로 뛴다"는 뜻이다. 모델 입장에서는 최선의 전략이 "애매하면 일단 PER/LOC라고 우기는 것"이 되어버렸고, 그 결과가 3단계에서 관찰한 precision 붕괴(PER precision 0.56→0.12)다.

### Run5 설계
O·ORG는 원래도 문제가 없었으니 가중치를 1.0(무보정)으로 고정해 오탐 페널티를 그대로 유지하고, PER·LOC만 ORG 총 토큰 수 대비 `sqrt(비율)`만큼 완만하게 올리되 최대 5배로 캡을 씌웠다. 실제 계산된 가중치: O 1.0 / B-ORG 1.0 / I-ORG 1.0 / B-PER 5.0 / I-PER 5.0 / B-LOC 5.0 / I-LOC 4.31.

구현: `scripts/train.py`에 `--class-weight-scheme {naive,capped}` 옵션 추가 (`naive`=Run4 방식 그대로 재현 가능, `capped`=Run5 방식). `compute_class_weights()` 함수 참조.

```
python scripts/train.py --run-name run5 --lr 2e-5 --epochs 3 --class-weight --class-weight-scheme capped
```

64건 스모크 테스트로 파이프라인 정상 동작 확인 후, 전체 데이터(143,292건)로 학습 실행. 약 2시간 12분 소요.

### 결과 — Run5 완료 (2026-07-22)

val set 기준:

| Entity | Precision | Recall | F1 | support |
|--------|-----------|--------|----|----|
| LOC | 0.66 | 0.92 | 0.77 | 238 |
| ORG | 0.91 | 0.94 | 0.93 | 7,045 |
| PER | 0.33 | 0.47 | **0.38** | 332 |
| **micro avg** | 0.87 | 0.92 | **0.8935** | 7,615 |

Run3(micro F1 0.9077, PER F1 0.47)와 비교하면 **더 나빠졌다.** 가중치를 최대 5배로 훨씬 완만하게 캡을 씌웠는데도, Run4와 똑같은 방향(recall 상승, precision 더 크게 하락)으로 무너졌다 — 다만 무너지는 정도는 Run4(PER F1 0.19)보다 약했다(PER F1 0.38). 즉 "가중치가 너무 세서" 실패한 게 아니라, **이 데이터에서는 class weight를 어떤 강도로 걸어도 PER/LOC F1이 개선되지 않는다**는 것을 두 번째로 확인한 셈이다.

### 최종 결론
class weight 접근(Run4 naive, Run5 capped)을 두 가지 방식으로 시도했지만 둘 다 실패했다. **Run3을 최종 baseline으로 유지**하며, PER의 약점(F1 0.47~0.55)은 클래스 불균형 자체보다 다른 요인(4단계에서 확인한 경계 오류, 원본 라벨 노이즈, 절대적으로 적은 학습 샘플 수 등)에 더 크게 기인하는 것으로 보인다. 이 결론을 6단계 한계점에 반영한다.

## 4단계 — 평가 (완료: 2026-07-21)

### 목표
3단계에서 선정한 Run3 모델을 한 번도 보지 않은 **test셋(9,026건)**으로 채점하고, entity별 F1과 오분류 패턴을 분석한다.

### 한 일
1. **평가 스크립트 작성** (`scripts/evaluate.py`): Run3 best_model로 test.jsonl 전체를 채점, seqeval classification report 산출. 이어서 앞부분 2,000건을 스캔해 예측과 정답이 다른 entity(놓친 것/오탐)를 모아 `results/run3/test_misclassified.json`에 저장.
2. **오분류 사례 분석**: 상위 200건을 직접 읽어서 패턴 분류.

### 결과 — test set 성능

| Entity | Precision | Recall | F1 | support |
|--------|-----------|--------|----|----|
| LOC | 0.76 | 0.89 | 0.82 | 291 |
| ORG | 0.90 | 0.94 | 0.92 | 6,573 |
| PER | 0.61 | 0.50 | **0.55** | 269 |

val F1(0.9077)과 test F1(0.9045)이 거의 같다 — **val에 과적합되지 않고 그대로 일반화됐다**는 뜻. 목표(0.80)를 확실히 넘었다.

### 오분류 사례 분석 (252건 중 상위 200건 기준)

전체 오탐/놓침의 대부분(168/200, 84%)이 **ORG**에서 나왔고, 세 가지 뚜렷한 패턴이 보인다:

1. **중첩된 기관명의 경계 오류 (가장 흔함)**: `국가정보원 심리전단`, `연세대학교 의과대학 세브란스병원`, `광주지방법원 본원 합의부`처럼 기관명이 여러 단어가 겹쳐 있을 때, 모델이 정답 전체 범위 대신 **앞부분(상위 기관)만 짧게 잘라서** 예측하거나(`국가정보원 심리전단` → `국가정보원`만), 반대로 뒤에 붙은 일반명사까지 **과하게 넓혀서** 예측한다(`연세대학교 의과대학 세브란스병원` → `...세브란스병원 소아 비뇨기과`까지 포함). 경계(boundary) 판단이 가장 약한 지점.
2. **"트위터"를 전혀 못 잡음**: 정답 라벨에는 `트위터`가 ORG로 여러 번 등장하는데 모델은 단 한 번도 잡지 못했다. 학습 데이터에서 SNS/플랫폼명이 ORG로 등장하는 빈도가 낮아서(대부분 법원·검찰·회사명 위주) 일반적인 고유명사 패턴과 다르게 생긴 이런 이름을 학습하지 못한 것으로 보인다.
3. **정답 라벨 자체의 노이즈 발견**: `다음과 같은 사실들이` 같은 문장에서 `다음`이 정답에 ORG로 라벨링된 사례가 있었다 — 실제로는 "아래(the following)"라는 뜻의 일반 단어이지 기관명이 아니다. 아마 포털 "다음(Daum)"과 같은 표기라 라벨링 규칙(또는 규칙 적용 스크립트)이 오탐한 것으로 보이는 **원본 데이터의 라벨링 오류**. 모델이 이 사례를 못 맞히는 건 모델 탓이 아니라 정답 자체가 틀렸을 가능성이 있다는 뜻.

PER·LOC은 상대적으로 사례가 적었지만(각 14~17건) 비슷하게 **경계 오류**(예: `아산시 선장면장` 중 `-장`(직책 접미사)을 포함할지 여부, placeholder `#이름#`을 `#`과 `이름`으로 잘못 나눠 잡는 경우)가 대부분이었다.

### 공식 벤치마크와 비교
- 공식 GPT-OSS-120B(QLoRA): F1 **0.9589** (AI Hub 비공개 test셋)
- 본 프로젝트 KLUE-BERT(Run3): F1 **0.9045** (자체 분리한 test셋)
- 차이(0.0544)의 원인: (1) 모델 크기가 120B 대 110M으로 1000배 이상 차이, (2) test셋이 다름(공식은 비공개 hidden set) — 직접 비교는 참고용, (3) 위에서 확인한 **기관명 경계 오류**가 실제 F1을 깎는 가장 큰 요인으로 보이는데, 이는 대형 모델일수록 문맥을 더 길게 반영해 중첩 기관명의 경계를 더 잘 잡을 가능성과 관련이 있어 보임(직접 검증은 못 했음).

**목표 F1 0.80을 test셋 기준 0.9045로 확실히 초과 달성.**
